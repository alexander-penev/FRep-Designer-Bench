# FRep Designer 4.0 — Examples Gallery

Eleven showcase scenes that demonstrate the range of features available
in FRep Designer. Every image here was generated by `frep_gallery`
(source: `tools/gallery.cpp`). Each scene fits in 30–50 lines of C++
and is a useful copy-paste starting point.

The gallery renders on the GPU_GLSL path when a GPU is available and on
the CPU_IR path otherwise. Because the two paths are visually equivalent
(mean |Δ| ~0.0008 — see `MULTIPATH.md`), the images are the same either
way; the ones committed here were rendered with the corrected shading
(per-light colour, NDC sky gradient, cancelled-form specular, analytic
dual-number normals, aligned raymarch).

Two scenes (06, 10) use image textures, which are a GPU-only material
feature — the CPU JIT can't sample them — so those are rendered on the GPU
and the committed images are kept when regenerating on a CPU-only host.

To regenerate the gallery yourself:

```bash
mkdir -p gallery
./build/frep_gallery gallery
```

## 01 — CSG operations

![CSG](gallery/01_csg.png)

Three CSG operations side by side:

- **Orange (left)** — `Difference(Sphere, Box)`. A cube has been
  subtracted from a sphere, leaving a clean cubical notch.
- **Green (centre)** — `SmoothUnion(Sphere, Sphere)` with `k=0.3`.
  Two spheres blend smoothly into a peanut/blob shape — the larger
  `k`, the smoother the join.
- **Blue (right)** — `Intersection(Sphere, Box)`. The result is a
  rounded cube (only the inside of both shapes).

Render time: ~160 ms on llvmpipe at 800×500.

## 02 — Non-linear deformations

![Deformations](gallery/02_deformations.png)

Three deformation primitives applied to box primitives:

- **Red (left)** — `TwistY` with `k=2.0`. The box rotates around the
  Y axis at a rate proportional to height, giving the helical look.
- **Green (centre)** — `BendXY` with `k=0.4`. Curves the geometry
  around the X axis.
- **Blue (right)** — `TaperY` with `t=0.25, h=1.6`. Linearly narrows
  the geometry from base to top.

All three deformations include Lipschitz correction so the SDF stays
valid (ray-marching converges cleanly without the typical "ghost
surface" artefacts).

Render time: ~430 ms — deformations are computationally heavier than
plain primitives because of the per-step coordinate transformation.

## 03 — Imported mesh, carved with CSG

![Mesh carve](gallery/03_mesh_carve.png)

A unit-sphere mesh (generated via marching cubes at 24³ resolution)
was voxelised into a `MeshSDFNode` at 48³ grid resolution, then
combined with a box via `Difference`. The result: a voxelised
sphere with a clean cubic notch carved out.

You can see the marching-cubes triangle facets on the sphere's
surface — these come from the 24³ MC resolution and could be smoothed
by increasing it. The carved face is perfectly flat because that
side comes from the box's analytic SDF, not the voxel grid.

This pattern (import mesh → voxelise → CSG) lets you bring external
geometry into a fully procedural pipeline.

Render time: ~366 ms (mesh storage buffer is large and bandwidth-
limited on the CPU-side Vulkan driver).

## 04 — Procedural patterns

![Patterns](gallery/04_patterns.png)

The five procedural pattern types, one per sphere:

| Pattern | Description |
|---|---|
| **Solid** (red) | Just `albedo`; no procedural component. |
| **Checker** (black/white) | 3D parity-based checkerboard — same algorithm Larry Gritz showed in 1987. |
| **Stripes** (purple/yellow) | Y-axis stripes, alternating `albedo` and `albedo2`. |
| **GradientY** (blue→orange) | Smooth `clamp` gradient along Y. |
| **Noise** (green) | Murmur-style integer-hash value noise. |

All patterns are evaluated in world space, so they "stick" to surfaces
naturally. The same code paths execute on CPU and GPU — when the GPU
path is active, the patterns are emitted directly into the GLSL
scene shader.

## 05 — CPU vs GPU comparison

| CPU (Cook-Torrance PBR) | GPU (Phong) |
|---|---|
| ![CPU](gallery/05_cpu.png) | ![GPU](gallery/05_gpu.png) |

Same scene rendered through both pipelines. The differences:

- CPU uses Cook-Torrance microfacet PBR; the sphere shows a more
  subtle highlight and the cube's edges are slightly softer.
- GPU uses simple Phong; brighter and slightly more "plastic" feel.

Timings (800×500):

- CPU JIT: 622 ms (single-threaded for this scene since it's small)
- GPU compute: 96 ms (~6.5× faster)

On a real GPU the speedup would be much larger; here the GPU runs on
llvmpipe (Mesa's software Vulkan driver) which is itself CPU-bound.

## 06 — Triplanar texture mapping

![Texture](gallery/06_texture.png)

A 128×128 procedural texture (concentric rings + radial colour shift)
projected onto a sphere (scale = 1.0×) and a cube (scale = 3.0×) via
triplanar projection.

Texture mapping on SDFs is harder than on meshes because SDFs lack
native UV coordinates. Triplanar projection samples the texture three
times — once against each canonical plane (XY, YZ, XZ) — and blends
the samples by the surface normal's component weights. This produces
seamless coverage for any geometry at the cost of mild blending
artefacts where two normal components are roughly equal (visible as
diamond-shaped patches on the sphere).

The cube has clean texture display on its flat faces because there
only one normal component is non-zero.

## 07 — Hero showcase

![Hero](gallery/07_hero.png)

Everything in one scene:

- **Floor**: Checker-patterned plane (`pattern_scale = 1.5`).
- **Left**: A twisted box (`TwistY`, `k = 2.5`) with stripe shading
  (orange/purple).
- **Centre**: A `SmoothUnion` of two spheres (`k = 0.25`) with

- **Right**: A sphere with a cubic notch (`Difference`) shaded with

## 08 — Gyroid via CustomExpr

![Gyroid](gallery/08_gyroid_customexpr.png)

A triply-periodic minimal surface defined by a runtime expression,
thickened into a slab and clipped to a sphere:

```
abs( sin(2x)cos(2y) + sin(2y)cos(2z) + sin(2z)cos(2x) ) - 0.4
```

The expression string lives in a `CustomExprNode`, which both the CPU
codegen and the GLSL emitter translate through its `codegen()` /
`emit_glsl()` methods — no C++ recompilation needed to change the
surface. The shape is intersected with a bounding sphere via
`Intersection` so it stays finite.

This is the gyroid surface used in materials science and 3D printing
for lightweight scaffolds — useful for showing that the project handles
non-CSG implicit surfaces gracefully.

Render time: ~140 ms at 600×400.

## 09 — Capsule primitive

![Capsule](gallery/09_plugin_capsule.png)

A capsule — a vertical segment swept by a radius — whose SDF
`sqrt(x² + max(0,|y|−h)² + z²) − r` is written as a `CustomExprNode`
expression, so it needs no new node type or plugin. The same shape is
also available as a dynamically-loaded `.so` plugin (see
`plugins/dynamic/capsule_plugin.cpp`), which defines a `Capsule` node
with its own `codegen()` and `emit_glsl()` — demonstrating that
third-party authors can ship both CPU and GPU support without modifying
the host emitter, the model renderer shading languages like MaterialX or
OSL use.

Render time: ~210 ms (capsule + floor) at 400×300.

## 10 — Wood + marble textured scene

![Textured scene](gallery/10_textured_scene.png)

Two procedural image textures sampled through the `Texture` material
pattern:

- Wood sphere: concentric-ring grain with a warm brown gradient,
  `pattern_scale = 1.0`
- Marble cube: sinusoidal veins broken up by noise,
  `pattern_scale = 2.0`

Image textures are a GPU-only material feature (the CPU JIT can't sample
them), so this scene renders on the GPU path. The texture pixels are
generated procedurally in `scene_textured()`; scenes can equally
reference texture files by path (`texture_path`), which travel alongside
the JSON and are reloaded automatically by `frep_multipath --paths gpu_glsl`.

Render time: ~120 ms at 600×400.

## 11 — Instancing, non-uniform scale, multi-axis rotation

![Instances and transforms](gallery/11_instances.png)

One authored shape reused six times, plus the new transform nodes:

- **Front row** — a twisted rounded box authored once (the red source on
  the left), then five blue **instances** of it. An instance references
  the source geometry rather than copying it (`InstanceNode` shares the
  pointer), so editing the source would update every instance live, and
  the emitted GLSL carries the shape's body once and calls it (Level 2
  shared subprogram: ~60% smaller code than six copies at higher counts).
- **Back row** — three ellipsoids, each a sphere under a **non-uniform
  `Scale`** stretched along a different axis (X, Y, Z). Non-uniform scale
  is not distance-preserving, so the SDF is scaled by the smallest axis
  factor to stay conservative.
- **Right** — a cube tilted on all three axes, stacking the new
  **`RotateX`** and **`RotateZ`** nodes with `RotateY`.

This single scene exercises instancing (both the reference semantics and
the shared-subprogram memory optimisation), per-axis scale, and all three
rotation axes together. Source: `scene_instances()` in `tools/gallery.cpp`.

Render time: ~940 ms (11 objects) at 1000×420.
